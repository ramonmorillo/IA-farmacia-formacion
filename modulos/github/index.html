M6 - GitHub y GitHub Pages para Farmacéuticos

Cómo guardar, publicar y actualizar una web sin saber programar

Inicio › GitHub y GitHub Pages

🎯 Objetivos de aprendizaje

  • Entender qué es GitHub y para qué sirve en este contexto
  • Crear un repositorio y subir los archivos de una web
  • Publicar la web en internet con GitHub Pages (gratis)
  • Actualizar el contenido cuando haya cambios
  • Comprender qué es el versionado y por qué protege tu trabajo

Tiempo estimado: 30 minutos Sin conocimientos técnicos requeridos

¿Qué es GitHub?

GitHub es un servicio en la nube donde se guardan proyectos formados por archivos de texto (como los HTML y CSS de esta plataforma). Puedes pensar en él como una carpeta compartida inteligente que, además de guardar tus archivos, recuerda cada cambio que has hecho, quién lo hizo y cuándo.

💡

Analogía farmacéutica:

Imagina el libro de registro de una farmacia hospitalaria: cada entrada tiene fecha, responsable y detalle de lo que se hizo. Si algo sale mal, puedes consultar el historial y volver a un estado anterior. GitHub hace lo mismo con los archivos de tu web: nada se pierde nunca y todo queda trazado.

Tres conceptos que necesitas entender

Término Qué es Equivalente cotidiano
Repositorio La carpeta raíz del proyecto, con todo su historial El expediente completo de un paciente
Commit Un conjunto de cambios guardados con un mensaje descriptivo Una entrada fechada en el libro de registro
Rama (branch) Una copia de trabajo donde pruebas cambios sin tocar la versión publicada Un borrador antes de firmar el documento definitivo
⚠️

GitHub ≠ Git: Git es el sistema de control de versiones que funciona en tu ordenador. GitHub es el servicio web que usa Git por debajo. En este módulo trabajamos siempre desde la web de GitHub, sin instalar nada.

¿Qué es GitHub Pages?

GitHub Pages es una función gratuita de GitHub que convierte los archivos HTML de tu repositorio en una web accesible desde internet. No necesitas contratar hosting, no necesitas configurar servidores. Con dos clics, tu web tiene una dirección pública del tipo:

https://tunombredeusuario.github.io/nombre-del-repositorio/

Esta plataforma formativa funciona exactamente así: los archivos HTML y CSS están en un repositorio de GitHub, y GitHub Pages los sirve como web.

✓

Ventajas para un proyecto formativo como este:

  • Completamente gratuito para repositorios públicos
  • Sin fecha de caducidad: la web sigue activa mientras el repositorio exista
  • Cada vez que actualizas los archivos, la web se actualiza automáticamente en minutos
  • Historial completo de cada versión publicada

Paso 1 — Crear una cuenta en GitHub

→

Si ya tienes cuenta, salta al Paso 2.

  1. Ve a github.com y haz clic en Sign up
  2. Elige un nombre de usuario: será parte de la dirección de tu web (tunombredeusuario.github.io). Elige algo profesional y fácil de recordar
  3. Introduce tu correo y una contraseña
  4. Elige el plan Free (es suficiente para todo lo que necesitas)
  5. Verifica tu correo electrónico
⚠️

Sobre la privacidad: Un repositorio público significa que cualquiera puede ver tus archivos. Para contenido formativo sin datos de pacientes esto no es un problema. Si en algún momento quisieras que el repositorio fuera privado, GitHub ofrece repositorios privados en el plan gratuito, pero entonces GitHub Pages no estaría disponible de forma gratuita.

Paso 2 — Crear el repositorio

El repositorio es el contenedor de todo tu proyecto. Créalo vacío: subirás los archivos después.

  1. Una vez dentro de GitHub, haz clic en el botón verde "New" (o en el icono + de la esquina superior derecha → New repository)
  2. Rellena el formulario:
    • Repository name: elige un nombre sin espacios ni acentos, por ejemplo ia-farmacia-formacion
    • Description: opcional, pero útil. Ej: "Plataforma formativa sobre uso responsable de IA en farmacia"
    • Public (necesario para GitHub Pages gratuito)
    • Marca "Add a README file" — esto evita que el repositorio quede vacío y facilita el primer paso
  3. Haz clic en "Create repository"
✓

Ya tienes tu repositorio. Verás una pantalla con el archivo README.md y la URL del repositorio en la barra del navegador: github.com/tunombredeusuario/ia-farmacia-formacion

Paso 3 — Subir los archivos de la web

Ahora subes los archivos HTML, CSS y cualquier otro contenido de tu proyecto.

Opción A — Subida directa desde el navegador (recomendada para empezar)

  1. En tu repositorio, haz clic en "Add file" → "Upload files"
  2. Arrastra y suelta los archivos y carpetas de tu proyecto (el index.html, la carpeta css/, la carpeta modulos/, etc.)
  3. En la sección "Commit changes" (al final de la página):
    • Escribe un mensaje descriptivo: "Subida inicial de la plataforma formativa v1"
    • Deja marcado "Commit directly to the main branch"
  4. Haz clic en "Commit changes"
⚠️

Limitación: La subida desde el navegador funciona bien para proyectos pequeños, pero tiene un límite de unos 100 archivos por subida. Si tu proyecto tiene muchos archivos, es mejor hacerlo en varias subidas por carpetas (primero css/, luego modulos/, etc.) o usar la opción B.

Opción B — Usando GitHub Desktop (para proyectos grandes o actualizaciones frecuentes)

GitHub Desktop es una aplicación gratuita que facilita sincronizar tu carpeta local con el repositorio sin usar línea de comandos.

  1. Descarga GitHub Desktop desde desktop.github.com e instálalo
  2. Inicia sesión con tu cuenta de GitHub
  3. Haz clic en "Clone a repository" y selecciona el repositorio que acabas de crear — esto descarga el repositorio como carpeta en tu ordenador
  4. Copia los archivos de tu proyecto dentro de esa carpeta
  5. En GitHub Desktop verás todos los archivos nuevos listados como "cambios". Escribe un mensaje en el campo "Summary" y haz clic en "Commit to main"
  6. Haz clic en "Push origin" para subir los cambios a GitHub
💡

¿Cuándo usar cada opción? Para la subida inicial o cambios puntuales de uno o dos archivos, el navegador es suficiente. Si vas a hacer actualizaciones frecuentes o manejas muchos archivos, invierte 10 minutos en configurar GitHub Desktop: ahorrarás mucho tiempo a largo plazo.

Paso 4 — Activar GitHub Pages

Con los archivos subidos, basta con decirle a GitHub que sirva ese repositorio como web.

  1. En tu repositorio, haz clic en Settings (pestaña con engranaje, en la barra superior)
  2. En el menú lateral izquierdo, haz clic en Pages
  3. En la sección "Branch", despliega el selector y elige main
  4. En el selector de carpeta, deja / (root)
  5. Haz clic en Save
⏱

GitHub tarda entre 1 y 3 minutos en desplegar la web por primera vez. Cuando esté lista, aparecerá en la misma página de Settings → Pages el mensaje: "Your site is live at https://tunombredeusuario.github.io/ia-farmacia-formacion/"

Si recargas la página pasado ese tiempo y no aparece, espera un poco más y vuelve a recargar.

✓

Comprueba que funciona: abre la URL en una pestaña nueva. Deberías ver tu index.html renderizado como web. Si ves solo texto sin estilos, comprueba que las rutas a los archivos CSS en el HTML son relativas (no empiezan por / ni por C:\).

Paso 5 — Actualizar el contenido

Una vez publicada la web, cada actualización sigue el mismo ciclo: edita → guarda (commit) → la web se actualiza.

Actualizar un archivo desde el navegador

  1. Navega hasta el archivo que quieres editar dentro de tu repositorio (por ejemplo, modulos/fundamentos/que-es-ia.html)
  2. Haz clic en el icono del lápiz (esquina superior derecha del visor de archivos)
  3. Edita el contenido directamente en el editor de texto de GitHub
  4. Cuando termines, baja hasta "Commit changes", escribe un mensaje descriptivo (ej: "Actualiza ejemplo de warfarina en M1.1") y haz clic en "Commit changes"

Actualizar desde GitHub Desktop

  1. Edita los archivos en tu ordenador con cualquier editor de texto
  2. Abre GitHub Desktop: verás los cambios detectados automáticamente
  3. Escribe un mensaje en "Summary" y haz clic en "Commit to main"
  4. Haz clic en "Push origin"
⏱

Tras hacer un commit, GitHub Pages actualiza la web publicada en 1-3 minutos. Si abres la URL y no ves el cambio, espera un poco y recarga con Ctrl+Shift+R (fuerza recarga ignorando caché).

⚠️

Buenas prácticas para los mensajes de commit: Escribe siempre mensajes que describan el qué y el por qué, no el cómo. Un mensaje útil: "Añade advertencia sobre dosis en módulo M2 — revisado con ficha técnica AEMPS". Un mensaje inútil: "cambios" o "actualización".

El concepto de versionado: por qué cambia todo

El versionado es la práctica de guardar cada cambio con un identificador y un mensaje, de forma que puedas ver exactamente qué era el proyecto en cualquier momento de su historia y, si es necesario, volver a una versión anterior.

¿Qué problema resuelve?

Sin versionado, el flujo habitual es:

modulo_m2_final.html
modulo_m2_final_v2.html
modulo_m2_final_v2_revisado.html
modulo_m2_ESTA_SI.html

Con versionado, tienes un solo archivo (modulo_m2.html) y GitHub guarda su historial completo. Puedes ver qué cambió entre ayer y hoy, o entre la versión de enero y la de marzo.

Tres situaciones donde el versionado te salva

1

Cometiste un error y quieres deshacerlo

Actualizaste el módulo M3 y sin querer borraste una sección entera. Sin versionado: a reescribir de memoria. Con versionado: vas al historial del archivo en GitHub, ves la versión anterior, copias el texto que faltaba y lo restauras.

Cómo ver el historial: En GitHub, abre el archivo → haz clic en "History" (esquina superior derecha) → selecciona cualquier commit anterior para ver cómo era el archivo en ese momento.

2

Quieres saber qué cambió entre dos versiones

Alguien modificó el módulo M2 hace tres semanas. ¿Qué se cambió exactamente? En GitHub, en el historial, haz clic sobre cualquier commit: verás en verde lo que se añadió y en rojo lo que se eliminó, línea por línea.

3

Quieres publicar una nueva versión sin arriesgar la actual

Estás preparando la versión 3 del módulo de fundamentos pero no quieres que los usuarios la vean hasta que esté lista. Crea una rama: trabaja en ella, prueba los cambios y, cuando estés seguro, fusiónala con la rama principal. La web publicada no se toca hasta ese momento.

Cómo crear una rama en el navegador: En tu repositorio, haz clic en el selector de ramas ("main", en la esquina superior izquierda) → escribe un nombre nuevo (ej: version-3-fundamentos) → haz clic en "Create branch".

Versiones mayores: etiquetas (tags)

Cuando publicas una versión completa (v1, v2, v3...) es buena práctica crear una etiqueta (tag) en ese commit. Así puedes volver exactamente a "cómo era la web el día que lanzamos la versión 2" con un solo clic.

  1. En tu repositorio, haz clic en Releases (panel lateral derecho) → "Create a new release"
  2. En "Choose a tag", escribe v2.0 y selecciona "Create new tag"
  3. Añade un título ("Versión 2 – Marzo 2026") y una descripción con los cambios principales
  4. Haz clic en "Publish release"
💡

Las releases son públicas y tienen su propia URL permanente. Son útiles para compartir con otros formadores exactamente qué versión usaron en un taller concreto, o para documentar qué cambios se revisaron con los responsables del programa.

El flujo completo en una imagen

Cada actualización de la plataforma sigue siempre este ciclo:

1

Editas

Modificas los archivos HTML en tu ordenador o le pides los cambios a Claude Code

2

Revisas

Abres el archivo en el navegador y compruebas que el resultado es correcto antes de guardarlo

3

Commit

Guardas los cambios con un mensaje descriptivo. Este paso crea una entrada permanente en el historial

4

Push

Subes el commit a GitHub. En 1-3 minutos la web publicada refleja los cambios automáticamente

Errores frecuentes

✗

La web aparece sin estilos (solo texto negro sobre blanco)

Causa más frecuente: Las rutas a los archivos CSS en el HTML son absolutas o incorrectas para el entorno de GitHub Pages.

Solución: Comprueba que en el <head> de tus HTML la ruta al CSS empieza por ../../css/base.css o similar (relativa), nunca por /css/base.css ni por una ruta local de tu ordenador.

✗

La web no se actualiza tras hacer un commit

Causa: GitHub Pages tarda entre 1 y 5 minutos. También puede ser caché del navegador.

Solución: Espera 3-5 minutos y recarga con Ctrl+Shift+R. Si sigue sin actualizarse, ve a Settings → Pages y comprueba que el source sigue siendo main / root.

✗

Un enlace interno devuelve "404 Not Found"

Causa: El archivo enlazado no existe en el repositorio o la ruta tiene una errata (mayúsculas, guiones, extensión).

Solución: Comprueba en el repositorio que el archivo existe exactamente con ese nombre. GitHub Pages distingue entre mayúsculas y minúsculas: Modulos/ y modulos/ son directorios diferentes.

⚠️

Commitear archivos con datos sensibles

Todo lo que suba a un repositorio público es visible para cualquiera. Nunca subas archivos con datos de pacientes, contraseñas, claves de API ni información personal. Una vez en el historial de Git, eliminar algo completamente es complejo.
